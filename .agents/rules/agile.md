# Agile/Adaptive - Rules

## 1. Terminología y Estándares

- **OBLIGATORIO:** Utilizar terminología ágil (Product Backlog, Sprint/Iteración, Historia de Usuario, Story Points, Criterios de Aceptación, Retrospectiva).
- **PROHIBIDO:** Utilizar términos rígidos de fases predictivas (Línea Base de Alcance, Acta de Constitución formal, WBS estricta) a menos que se solicite un modelo híbrido.

## 2. Gestión de Requisitos

- Los requisitos nunca están "congelados". Se capturan como Historias de Usuario con el formato: *"Como [rol], quiero [acción] para [beneficio/valor]"*.
- La prioridad del Backlog la dicta siempre el valor que aporta al negocio, usando técnicas como MoSCoW (Must have, Should have, Could have, Won't have).

## 3. Estimación y Métricas

- Evitar estimar en horas o días exactos en etapas tempranas. Usar medidas relativas como Story Points (secuencia de Fibonacci) o Tallas de Camiseta (S, M, L, XL) para medir el esfuerzo.
- Medir el progreso mediante la "Velocidad" (puntos completados por iteración) y no por el porcentaje de presupuesto consumido.

## 4. Flujo de Trabajo y Entregas

- Fomentar prácticas de integración continua. Toda nueva característica debe planificarse pensando en su fusión segura con el entorno principal y su registro documental.
- Cada iteración debe producir un incremento de producto potencialmente entregable o funcional.
